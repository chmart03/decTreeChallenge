\subsection{The Decision Tree Problem
ðŸŽ¯}\label{the-decision-tree-problem}

\begin{quote}
``The most important thing in communication is hearing what isn't
said.'' - Peter Drucker
\end{quote}

\textbf{The Core Problem:} Decision trees are often praised for their
interpretability and ability to handle both numerical and categorical
variables. But what happens when we encode categorical variables as
numbers? How does this affect our understanding of feature importance?

\textbf{What is Feature Importance?} In decision trees, feature
importance measures how much each variable contributes to reducing
impurity (or improving prediction accuracy) across all splits in the
tree. It's a key metric for understanding which variables matter most
for your predictions.

\begin{tcolorbox}[enhanced jigsaw, colback=white, arc=.35mm, titlerule=0mm, opacitybacktitle=0.6, breakable, rightrule=.15mm, toprule=.15mm, coltitle=black, bottomrule=.15mm, colbacktitle=quarto-callout-important-color!10!white, bottomtitle=1mm, colframe=quarto-callout-important-color-frame, left=2mm, leftrule=.75mm, toptitle=1mm, title=\textcolor{quarto-callout-important-color}{\faExclamation}\hspace{0.5em}{ðŸŽ¯ The Key Insight: Encoding Matters for Interpretability}, opacityback=0]

\textbf{The problem:} When we encode categorical variables as numerical
values (like 1, 2, 3, 4\ldots), decision trees treat them as if they
have a meaningful numerical order. This can completely distort our
analysis.

\textbf{The Real-World Context:} In real estate, we know that
neighborhood quality, house style, and other categorical factors are
crucial for predicting home prices. But if we encode these as numbers,
we might get misleading insights about which features actually matter
most.

\textbf{The Devastating Reality:} Even sophisticated machine learning
models can give us completely wrong insights about feature importance if
we don't properly encode our variables. A categorical variable that
should be among the most important might appear irrelevant, while a
numerical variable might appear artificially important.

\end{tcolorbox}

Let's assume we want to predict house prices and understand which
features matter most. The key question is: \textbf{How does encoding
categorical variables as numbers affect our understanding of feature
importance?}

inherent order or magnitude relationship to house prices. These must be
modelled as categorical variables.

\subsection{Discussion Questions}\label{discussion-questions}

\subsubsection{1. Numerical vs Categorical
Encoding}\label{numerical-vs-categorical-encoding}

Zip code should definitely be treated as a categorical variable. Even
though it looks like a number, it doesn't actually behave like one.
There's no real meaning behind ``50012 is greater than 50011,'' and that
shouldn't imply anything about house prices. Treating it numerically
makes the tree create splits like zipCode \textgreater{} 50012.5, which
honestly means nothing in the real world. Neighborhoods matter, but the
number of the zip code itself has no built-in ordering.

So for the model to make sense, zipCode should be handled as a
non-ordinal categorical variable, not a number.

\subsubsection{2. R vs Python Implementation
Differences}\label{r-vs-python-implementation-differences}

Between the two, R handles categorical variables better out of the box.
In R, decision-tree packages like rpart already know how to deal with
categorical fields. They'll split by groups of categories (like certain
zip codes vs the rest), which makes a lot more sense.

In Python (specifically scikit-learn), decision trees don't natively
support categorical features. This is straight from the scikit-learn
docs:

\begin{quote}
``The scikit-learn implementation does not support categorical variables
for now.''
\end{quote}

Because of that, Python forces you to encode it first. And if someone
just passes the zip code in as an integer without encoding it, the model
will treat it like numeric data --- which is exactly how you end up with
useless splits and feature importances that don't reflect reality.

So for this specific variable, R naturally handles it the right way,
while Python needs extra work (and it's easy to get wrong).

\subsubsection{3. Suggestions for Implementing Decision Trees in Python
With Proper Categorical
Handling}\label{suggestions-for-implementing-decision-trees-in-python-with-proper-categorical-handling}

There are better ways to handle categorical features in Python besides
just slapping an integer on them.

\textbf{One-Hot Encoding (basic but safe for non-ordinal categories):}
This works well when the number of categories isn't too crazy. It
prevents the model from assuming an order. The downside is it creates a
bunch of new columns and can spread out the importance.

\textbf{Avoid plain label/ordinal encoding:} If you map categories to
numbers (like zip â†’ 1, 2, 3, \ldots), the tree will think those numbers
mean something. That's exactly what caused the problem in the challenge.
LabelEncoding turns categories into integers, which the tree treats as
numeric splits. If the data isn't ordinal, this is wrong. This is why
zipCode-as-numeric became misleading.

\textbf{Tree-based encoders (a smarter option):} Libraries like
feature\_engine have things like DecisionTreeEncoder, which actually
fits a small tree to each categorical variable and replaces each
category with a ``predicted value.'' Instead of arbitrary numbers, you
get an encoding that actually reflects how each category relates to the
target. This keeps the ``interpretability'' of a tree without making up
fake numeric relationships.

\textbf{Libraries that handle categoricals natively:} Tree libraries
like CatBoost and LightGBM (with specific categorical handling) do this
way better than scikit-learn and are built for these situations.
